## 0. 北极星（1–2 年）

先搭建"补充自己能力/认知"的系统，逐步长成可对外开放的**多用户 agent 平台**：
统一用户体系 · 按角色开放不同菜单 · 积分系统（站长管理）· 充值/支付。
最终服务于**变现**与 **OPC（一人公司）→ OPT（一人团队）**。

**贯穿原则——"platform-ready，但不提前造地基"**
每个子系统现在做成**独立模块、单用户能跑**；但从第一行代码就遵守统一约定（技术栈、API/数据约定、预留 `userId`、统一设计语言），将来抽出共享用户体系时是"插上去"，不是"推倒重来"。

## 1. 三条轨

| 轨 | 名称 | 现在状态 | 说明 |
|---|---|---|---|
| **A** | 能力扩充 | 🟢 主攻 | 建"补充认知/能力"的系统。首个子项目 = A1 信息站升级。 |
| **B** | 变现 | ⚪ 后置 | 对外开放、积分、充值、付费产品。等 P3/P4。 |
| **C** | IP/影响力 | 🟡 并行 | 公众号/知乎/抖音，边建设边打个人 IP（build-in-public）。 |

**A×C 协同闭环（关键）**：
A1 把每日资讯**消化成知识卡/洞见** → 这些洞见 + "我在搭平台"的建设过程，本身就是 C 轨的内容原料 →
所以 **A1 的产出必须"可导出为内容素材"**（这是 A1 的一条设计约束）。

**C 轨执行机制——两种内容，分清楚**：
1. **洞见内容**：A1 消化资讯产出的"AI产品经理解读 X"草稿（A1 负责）。
2. **建设日志（build-in-public）**：记录"我如何从零搭平台 / 走向 OPC"的实现路径，**面向同路人**（吸引正在路上的人）。**由 Claude 执笔**，每到里程碑写一篇草稿，存 `docs/journey/`。这是元层内容，不并入任何子项目。

## 2. 分阶段路线图

| 阶段 | 名字 | 内容 | 碰"钱/多用户"？ |
|---|---|---|---|
| **P0** | 约定先行 | 定 monorepo 结构、技术栈基线、API/数据/命名约定、未来用户体系蓝图（**只写文档**） | 否 |
| **P1** | 能力系统 MVP | **A1：信息站升级（从"看"到"消化"）**，单用户自用 | 否 |
| **P2** | 第二能力模块 | 再加一个模块，验证"约定"能否让两模块共存 | 否 |
| **P3** | 整合骨架 | 抽出统一用户/认证/菜单权限（哪怕只你一人） | 多用户雏形 |
| **P4** | 对外+变现 | 开放注册、积分、充值、上域名/服务器（进入 B 轨） | **是** |

C 轨贯穿 P0–P4：每完成一个里程碑，沉淀 1 篇内容。

## 4. 技术栈基线（草案，P0 细化）

- 语言/运行时：Node.js 22.5+（已用内置 `node:sqlite`）。
- 前端：React + Ant Design + Vite（信息站已是）。
- 数据：起步 SQLite，整合期评估迁移 Postgres。
  - **DB 迁移边界（决策）**：所有 SQL 关在各模块的 `*-db.js` 里，路由/前端只调函数。换 Postgres = 只重写该文件。现在**不**上 ORM（单用户 MVP 过度设计）。真正的多用户前向兼容靠 `user_id` 贯穿，已落实。
- AI：OpenAI 协议网关（可接 Claude），统一封装，便于换模型/控成本。
- 预留：所有业务表预留 `user_id`；API 走 `/api/<模块>/...` 命名空间。

## 5. 盲点雷达（伙伴提醒 · 走到对应阶段重点补）

- **P4 收钱前**：支付合规、发票、法律主体（OPC 注册）、用户协议/隐私政策。
- **P3 多用户**：认证安全、权限越权、数据隔离、攻击面。
- **P4 上线**：生产级运维 ≠ 会装 docker —— 备份、监控、容灾、回滚。
- **全程**：AI token 成本随用户增长的失控风险，需限额/计费设计。
- **C 轨**：内容产出的可持续节奏（别为更新而透支建设时间）。
